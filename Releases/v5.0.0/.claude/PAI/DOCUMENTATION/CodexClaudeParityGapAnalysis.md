# PAI Claude Code ↔ Codex Parity — Gap Analysis

Scope: implemented feature parity for the PAI ecosystem across the two target CLIs,
**Claude Code** and **Codex**. This report records the architecture both CLIs share,
the per-capability parity status with end-to-end evidence, the gaps found and closed in
this pass, and the one environmental (non-parity) blocker on the current Windows box.

- Branch: `pai-provider-parity-gaps`
- Source of truth (release tree): `Releases/v5.0.0/.claude`
- Live installs verified: `~/.codex` (active) and `~/.claude`, shared data root `~/.pai`
- Codex CLI: `codex-cli 0.142.0` · Bun 1.3.14 · Windows 11

## 2. Architecture: how parity is achieved

PAI uses a **single shared bundle + generated per-CLI adapters** pattern, not a forked
per-CLI tree. This is the existing repo approach; this pass extends it, it does not replace it.

```
            Releases/v5.0.0/.claude         (release source of truth)
                       │  install / framework switch (PAI-Install + pai.ts)
        ┌──────────────┴───────────────┐
   ~/.claude (Claude Code)        ~/.codex (Codex)
   - settings.json (hooks)        - config.toml (root + MCP + hooks.state)
   - CLAUDE.md                    - AGENTS.md (generated from CLAUDE.md)
   - commands/*.md                - prompts/*.md (generated from commands)
   - agents/*.md                  - agents/*.toml (generated from agents/*.md)
   - PAI/* (symlink → release)    - PAI/* (symlink → release)
        └──────────────┬───────────────┘
                       │  MEMORY + USER symlinks
                  ~/.pai/{MEMORY,USER}        (shared Life-OS state)
                  ~/.pai/framework.json       (active framework pointer)
```

Key facts (verified on disk this pass):

- `~/.codex/PAI/{TOOLS,PAI-Install,DOCUMENTATION,bin,ALGORITHM,TEMPLATES}` are **symlinks**
  to the release tree — runtime tool fixes propagate to the install with no reinstall.
- `~/.codex/{hooks,skills,commands,agents,prompts}` and `PAI/PULSE` are **materialized copies**,
  (re)generated by the installer / `pai framework switch`.
- `~/.codex/MEMORY`, `~/.codex/USER`, `~/.codex/PAI/MEMORY`, `~/.codex/PAI/USER` and the same
  four under `~/.claude` all resolve (realpath) to `~/.pai/{MEMORY,USER}` — one physical store.
- `~/.pai/framework.json` = `{ active: "codex", root: ~/.codex, dataDir: ~/.pai }`.
- Path resolution centralizes in `PAI/TOOLS/lib/paths.ts` + `lib/transcripts.ts`
  (`getPaiDir`, `getDataDir`, `getFrameworkDir`, `getActiveFramework[Root]`), so the same code
  resolves the active framework files separately from shared user data.

Generation entry points:
- `PAI-Install/engine/config-gen.ts` — `generateCodexConfigToml`, `mergeCodexConfigToml`,
  `generateCodexHooksJson`.
- `PAI-Install/engine/actions.ts` — `syncCodexPrompts`, `codexPromptContent`, `syncCodexAgents`.
- `PAI/TOOLS/pai.ts` — runtime launcher, `framework switch`, MCP profile application
  (`buildCodexMcpConfigToml`, `writeCodexConfigToml`).

This satisfies the objective's "single source of truth OR kept-in-sync CLI adapters" — PAI
uses both: one shared bundle + deterministically generated, kept-in-sync Codex artifacts.

## 3. Capability-by-capability detail + evidence

### 3.1 Hooks — ✅
- **Implementation:** `generateCodexHooksJson` emits a Codex `hooks.json` covering the full PAI
  lifecycle (SessionStart, PreToolUse, PostToolUse, UserPromptSubmit, PreCompact, PostCompact,
  Stop, StopFailure, PostToolUseFailure, SessionEnd). Each entry runs
  `FrameworkHookAdapter.ts --framework codex --target <hook(s)>`, which reads Codex's hook JSON
  from stdin and normalizes event/tool/input/result/prompt/cwd/transcript/session fields into the
  Claude-shaped contract existing PAI hooks expect. Hook commands embed `PAI_DIR`, `PAI_DATA_DIR`,
  `PAI_FRAMEWORK`, `PAI_FRAMEWORK_DIR`, `PAI_SETTINGS_PATH`, `PAI_CONFIG_DIR`, and use the Windows
  `&` call operator with direct `bun.exe` invocation.
- **Evidence (live):** `CodexRealSessionHookProof` ran a real `codex exec`; the model executed a
  shell command and PAI's PostToolUse `ToolActivityTracker` recorded the unique marker into
  `~/.pai/MEMORY/OBSERVABILITY/tool-activity.jsonl`. The AGENTS.md context run additionally showed
  `hook: Stop` / `hook: Stop Completed` (Stop-phase hooks firing). Codex's `config.toml [hooks.state]`
  carries per-hook `trusted_hash` entries for `session_start/pre_tool_use/post_tool_use/`
  `user_prompt_submit/stop/pre_compact/post_compact` — proof Codex itself loaded and trust-evaluated
  the hooks.

### 3.2 MCP servers — ✅
- **Implementation:** PAI ships 9 MCP profiles under `MCPs/`. `pai mcp set <profile>` /
  `pai -m <names>` write the selected servers into Codex's native `config.toml` between
  `# BEGIN/END PAI MANAGED MCP CONFIG`, emitting `[mcp_servers.<name>]` tables with
  `url`/`command`/`args`/`env`/`env_vars`/timeouts and preserving any user-defined servers.
- **Evidence (live):** Codex natively loads `[mcp_servers.*]` — the live `~/.codex/config.toml`
  has a working `[mcp_servers.node_repl]` that is actively running in the Codex runtime. `pai mcp list`
  resolves against the live Codex install (6 profiles). `PaiDoctor` asserts the managed MCP block is
  present in the live config. MCP profiles parse with no literal secrets (security audit green).

### 3.3 Agents / subagents — ✅
- **Implementation:** `syncCodexAgents` renders each shared `agents/*.md` into a Codex-native
  `agents/<slug>.toml` (`name`, `description`, `developer_instructions`) via the provider-neutral
  agent contract (PAI path bootstrap, AGENTS.md startup context, `$PAI_DATA_DIR` memory roots,
  no Claude-only references). Autonomous/parallel work uses a shared framework-agent launcher that
  drives `codex exec` (workspace-write, `--skip-git-repo-check`, `--cd`, prompt on stdin).
- **Evidence:** 18 `agents/*.toml` generated in the live install. `CodexFrameworkAgentExecutionSmokeTest`
  is a runtime proof (fake recording `codex` shim) that the real launchers invoke `codex exec` with the
  contracted subagent flags — green in the deep run. `FrameworkSmokeTest --framework codex` asserts
  generated agents are provider-neutral and carry the `$PAI_DIR` fallback bootstrap.

### 3.4 Context / config loading — ✅
- **Implementation:** `CLAUDE.md` → `AGENTS.md` (path/term rewrites: `CLAUDE.md`→`AGENTS.md`,
  `Claude Code`→`Codex`, `~/.claude/PAI`→`$PAI_DIR`, `~/.claude`→`$PAI_FRAMEWORK_DIR`). Codex loads it
  plus `RTK.md` via `config.toml project_doc_fallback_filenames = ["AGENTS.md","RTK.md","CLAUDE.md"]`.
  The `LoadContext` SessionStart hook injects dynamic PAI context on top.
- **Evidence (live):** A `codex exec` session reproduced the exact first heading of its loaded
  instructions — `# PAI 5.0.0 — Personal AI Infrastructure (the Life Operating System)` — confirming
  Codex read the generated AGENTS.md. `LoadContextSmokeTest` and `PaiDoctor` (root block + fallback
  docs) green.

### 3.5 Slash commands — ✅
- **Implementation:** `syncCodexPrompts` generates Codex `prompts/*.md` from shared `commands/*.md`,
  converting Claude skill redirects to Codex syntax (`Skill("X","$ARGUMENTS")` → `$X $ARGUMENTS`).
  Because the installed Codex 0.142.0 TUI does not surface `/prompts:<name>` in its slash menu, the
  stable contract is direct `$Skill` mention syntax; the generated prompts remain a compatibility bridge.
- **Evidence (live):** All four commands (`cs`, `context-search`, `pu`, `interview`) have matching
  `prompts/*.md`; every `Skill()` call is converted to `$Skill` with no residual `Skill("` calls and a
  `generated for Codex` provenance marker.

### 3.6 Shared memories across CLIs — ✅
- **Implementation:** `~/.pai/{MEMORY,USER}` is the single Life-OS store. The installer/`pai framework
  switch` link `MEMORY`/`USER` and `PAI/MEMORY`/`PAI/USER` in **both** `~/.codex` and `~/.claude` to it,
  and write `~/.pai/framework.json`.
- **Evidence (live):** The marker written by the Codex-session hook (3.1) is present in the shared
  observability log, and `~/.pai/...`, `~/.codex/PAI/MEMORY/...`, and `~/.claude/PAI/MEMORY/...` all
  `realpath` to the **same file** — a Codex write is instantly visible from the Claude home.
  `FrameworkSmokeTest --framework codex` "codex sees shared memory marker" / "codex sees shared user
  marker" green. Mnemopi long-term memory (`~/.codex/memories_1.sqlite` etc.) is Codex-app-native and
  orthogonal to PAI's shared MEMORY tree.

CLI swap friction: `pai framework switch claude|codex` flips `framework.json` and regenerates native
artifacts while preserving `~/.pai`. Mid-session live state hand-off is explicitly out of scope per the
objective; shared durable memory/user state is the requirement and it holds.

## 4. Gaps found and fixed in this pass

### 4.1 Codex sessions mis-tagged as OpenCode in Pulse cost aggregation — FIXED (real bug)
- **Symptom:** `PerformanceCostAggregatorSmokeTest` failed — a seeded Codex transcript produced no
  cost row ("1 ignored").
- **Root cause:** `getActiveFrameworkRoot(framework)` (`lib/transcripts.ts`) ignored its `framework`
  argument when `PAI_FRAMEWORK_DIR` was set, returning the active dir for *every* framework. The
  cost-aggregator iterates `[claude, codex, opencode]`; codex and opencode both resolved
  `<root>/sessions` to the same directory, and last-wins dedup re-tagged Codex sessions as `opencode`.
  The opencode branch needs `usage` fields Codex transcripts lack → the row returned null → ignored.
  Real-world impact: a Codex user's session-cost telemetry would be silently dropped.
- **Fix:** added `frameworkRootFor(framework)` — the active framework still honors
  `framework.json`/`PAI_FRAMEWORK_DIR`; non-active frameworks resolve to their own conventional homes
  (`CODEX_HOME`/`~/.codex`, `CLAUDE_HOME`/`~/.claude`, `OPENCODE_CONFIG_DIR`/`~/.config/opencode`), so
  cross-framework scans never collide. Only `cost-aggregator` iterates non-active frameworks;
  `ActivityParser`/`SessionHarvester`/`getSessionFiles` use the active framework and are unchanged.
- **File:** `PAI/TOOLS/lib/transcripts.ts`. **Verify:** `PerformanceCostAggregatorSmokeTest` green;
  `TranscriptParserSmokeTest`, `ProviderTranscriptConsumersSmokeTest`, `FrameworkSmokeTest --framework
  codex` green (no regression).

### 4.2 SkillGuard hard-block not allowlisted in security audit — FIXED
- **Symptom:** `PaiSecurityAuditSmokeTest` flagged `hooks/SkillGuard.hook.ts` as an un-allowlisted
  hard-block surface.
- **Root cause:** `SkillGuard` legitimately emits `permissionDecision: "deny"` for known
  false-positive skills (mirrors Claude's Pulse skill-guard for Codex command hooks) but was not in
  `ALLOWED_NONZERO_HOOKS`.
- **Fix:** added `hooks/SkillGuard.hook.ts` to the allowlist with a rationale comment.
- **File:** `PAI/TOOLS/PaiSecurityAuditSmokeTest.ts`. **Verify:** "hook hard-block surfaces
  allowlisted" green.

### 4.3 Generated Codex hooks missing from hotfix manifest — FIXED
- **Symptom:** `PaiSecurityAuditSmokeTest` "hotfix manifest covers generated Codex hook targets"
  failed for `RestoreContext`, `SkillGuard`, `StopFailureHandler`, `ToolFailureTracker` — these are
  emitted into Codex `hooks.json` but absent from `hotfix-manifest.json`, so a hotfix update would not
  refresh them in installed homes.
- **Fix:** added the four hook entries to `hotfix-manifest.json`.
- **File:** `hotfix-manifest.json`. **Verify:** "hotfix manifest covers generated Codex hook targets"
  green.

### 4.4 Updater smoke tests wrote the real OneDrive PowerShell profile — FIXED
- **Symptom:** with `update-installed.ps1` restored, `JunctionSafeUpdateSmokeTest` (scenario 1/3) and
  `HotfixUpdateRollbackSmokeTest` (offline + opencode invocations) failed with PowerShell `Set-Content`
  `UnauthorizedAccessException` writing `…\OneDrive\…\profile.ps1`.
- **Root cause:** the updater's profile repair enumerates `$env:OneDrive` profiles (intended for real
  installs), but those two test invocations did not isolate `OneDrive` (only the first invocation did),
  so they inherited the machine's real `$env:OneDrive` and tried to rewrite the user's actual OneDrive
  PowerShell profile — blocked by Controlled Folder Access / AV → updater aborted (exit 1), cascading
  into downstream assertions.
- **Fix:** set `OneDrive` to a per-test temp dir in every updater invocation
  (`HotfixUpdateRollbackSmokeTest.ts` offline + opencode envs; `JunctionSafeUpdateSmokeTest.ts`
  `runUpdater`), matching the first invocation's existing isolation, so the tests are hermetic and never
  touch the real profile.
- **Files:** `PAI/TOOLS/HotfixUpdateRollbackSmokeTest.ts`, `PAI/TOOLS/JunctionSafeUpdateSmokeTest.ts`.
  **Verify:** `JunctionSafeUpdateSmokeTest` and `HotfixUpdateRollbackSmokeTest` fully green.
- **Recommended follow-up (not applied — AV, see §5):** `Repair-PowerShellProfiles` in
  `update-installed.ps1` should wrap each profile write in try/catch so one CFA/AV-blocked profile warns
  and continues instead of aborting the whole update — a real robustness gap for end users on
  OneDrive/CFA machines. It could not be applied here because AV blocks writing modified
  `update-installed.ps1` content on this box.

### 4.5 OpenCode hotfix regeneration wrote to the wrong root and skipped Claude agents — FIXED
- **Symptom:** `HotfixUpdateRollbackSmokeTest` OpenCode checks failed — after `update-installed.ps1
  --framework opencode`, `opencode.json` kept `instructions:["CLAUDE.md"]` plus `env`/`pai` keys, and
  `agents/*.md` / `commands/cs.md` stayed Claude-style (not provider-neutral, still had `Skill()` /
  Claude-only frontmatter).
- **Root cause (two bugs in `pai.ts`):**
  1. The regen runs `bun pai.ts framework switch opencode` *from the install it regenerates*, so
     `CURRENT_INSTALL_ROOT === target`. `frameworkRoot()`'s `wouldContaminateCanonicalSource` guard then
     rejected the install's `OPENCODE_CONFIG_DIR` and fell back to the default `$HOME/.config/opencode`,
     writing native artifacts to the wrong directory and leaving the install stale.
  2. `syncOpenCodeAgents()` only converted agents when the agents dir was a symlink (fresh install) or
     the file already carried a PAI / `Skill(` / `argument-hint:` marker. A real (copied) dir of raw
     Claude agents — which have none of those markers — was skipped, so agents never became native.
- **Fix:**
  1. `frameworkRoot()` now honors the framework-specific home env (`CODEX_HOME` / `OPENCODE_CONFIG_DIR` /
     `CLAUDE_HOME`) as a deliberate target even when it equals `CURRENT_INSTALL_ROOT`; the
     canonical-source guard is kept only for the generic `PAI_FRAMEWORK_DIR` override.
  2. `syncOpenCodeAgents()` now regenerates any agent that is not already a native OpenCode agent
     (`mode: subagent`), converting raw Claude-style agents in place while preserving genuine
     user-authored native agents.
- **File:** `PAI/TOOLS/pai.ts`. **Verify:** all 6 OpenCode checks in `HotfixUpdateRollbackSmokeTest`
  green; `FrameworkSmokeTest --framework codex`, framework command/launch smokes, and
  `OpenCodeFrameworkAgentExecutionSmokeTest` green (no regression to Claude/Codex resolution).

## 5. Environmental blocker (NOT a parity gap): `update-installed.ps1`

On this Windows box, the Windows hotfix updater `update-installed.ps1` (a ~37 KB self-updating
PowerShell downloader) was **quarantined by local antivirus** — `git status` showed it deleted from the
working tree though tracked in git, and any operation recreating the file at that path was denied
(`EPERM`) regardless of content, while other filenames in the same directory wrote fine. **The user
cleared the quarantine and the file was restored from git `HEAD`** (38,342 bytes, verified); the
`PAI security audit` and `CodexNativeRuntime` smokes that read it now pass.

This was **environmental, not a parity defect**: the file is intact in git, ships in CI (the
`PAI Codex Validation` workflow is green on ubuntu/macos/windows), and is present in the installed
`~/.codex/update-installed.ps1`. It is the *update mechanism*, not one of the six target capabilities.

Residual AV constraint: editing `update-installed.ps1` *in place* is still blocked on this machine — a
plain `git restore` of the exact original blob succeeds, but writing any *modified* content is re-flagged
(content/hash-based AV). This only affects the recommended `Repair-PowerShellProfiles` hardening in
§4.4; it does not affect any of the six capabilities.

## 6. Known boundaries (pre-existing, not parity regressions)

- The release source directory is still named `.claude` for v5.0.0 packaging history; framework-native
  install logic adapts it into Codex/OpenCode homes.
- Some command/agent definitions intentionally remain shared Claude-style markdown source; Codex
  receives generated `prompts/*.md` and `agents/*.toml` with Codex-native syntax.
- Optional external MCP integrations remain warnings (not failures) when API tokens are absent.
- Live mid-session context/state hand-off between CLIs is not implemented (explicitly out of scope);
  durable shared memory/user state across CLIs is implemented and verified.
- **OpenCode** is a third framework in the repo. At the user's request its update path was brought to
  green too: the hotfix regeneration (`opencode.json` instruction/key rewrite, native agent/command
  generation via `bun pai.ts framework switch opencode`) is fixed (§4.5) and the full OpenCode hotfix
  scenario passes. Full OpenCode *capability* parity (the same six capabilities) remains a larger,
  separate effort beyond this pass; only the regeneration/update path was addressed here.
